# zk-lsp

A Language Server Protocol server for a [Typst]-based [Zettelkasten] wiki.

Build a personal wiki in Typst — rich typesetting, programmable templates,
compiled to PDF — while `zk-lsp` keeps your note graph consistent and your
editor experience rich.

[Typst]: https://typst.app
[Zettelkasten]: https://en.wikipedia.org/wiki/Zettelkasten

## What you get

- **Inlay hints** — `@2602082037` renders inline as `@ Note Title`
- **Diagnostics** — dead links, archived references, dependency cycles, schema errors
- **Code actions** — toggle `checklist-status`, switch `relation`
- **Completions** — TOML enum values, note IDs, field names
- **References / go-to-definition** — jump between linked notes instantly
- **Hover** — preview linked note content without leaving the current buffer
- **Format pipeline** — save-time Lua hooks normalise checkboxes and
  compute derived metadata
- **Reconcile DSL** — a Lisp-family rule language propagates done-states
  across the wiki graph without touching Rust
- **CLI tools** — `new`, `remove`, `metadata`, `generate`, `export`,
  `check`, `reconcile`, `note-info`

## Quick Start

```bash
# 1. Create a new wiki
mkdir ~/my-wiki && cd ~/my-wiki
zk-lsp init

# 2. Open the sample note
nvim $(ls note/*.typ | tail -1)

# 3. Create notes
nvim $(zk-lsp new)

# 4. Compile the full wiki
typst compile index.typ
```

`init` creates:

```
my-wiki/
├── include.typ   # template definitions — edit freely
├── index.typ     # entry point — add a TOC or cover page here
├── link.typ      # auto-generated by zk-lsp — do not edit
├── metadata.toml # central metadata index — manage via zk-lsp metadata
└── note/
    └── <timestamp>.typ
```

## Note and Metadata Shape

A note owns its ID, title, body, links, and checkboxes. It only indexes into
central metadata:

```typst
#import "../include.typ": *
#let zk-metadata = zk_metadata("2607081902")
#show: zettel.with(metadata: zk-metadata)

= Example Title <2607081902>

Body text with @2607081903 references.
```

`metadata.toml` owns all metadata fields for every note. Prefer
`zk-lsp metadata` commands for ordinary changes instead of hand-editing this
index:

```toml
format-version = 1

[notes."2607081902"]
schema-version = 1
aliases = []
abstract = ""
keywords = []
generated = true
checklist-status = "none"
relation = "active"
relation-target = []

[notes."2607081902".user]
project = ""
```

Common flows:

```text
zk-lsp new
    |
    +--> note/ID.typ
    |      #let zk-metadata = zk_metadata("ID")
    |      = Title <ID>
    |
    +--> metadata.toml
           [notes."ID"] complete default record

zk-lsp format < note/ID.typ
    |
    +--> stdout: formatted note body
    |
    +--> metadata.toml: current note's record patched by hooks

zk-lsp metadata set ID relation=archived
    |
    +--> metadata.toml: current note's complete record updated

editor navigation
    |
    +--> note title/binding ID  ----definition---->  [notes."ID"]
    |
    +--> metadata record key    ----definition---->  note title
    |
    +--> references include body @ID, record key, and relation-target IDs
```

## Neovim Integration

```lua
-- lsp/zk-lsp.lua (Neovim 0.11+)
return {
  cmd = { "zk-lsp", "lsp" },
  filetypes = { "typst" },
  root_dir = function(bufnr)
    return vim.fs.root(bufnr, { "zk-lsp.toml", ".git" }) or vim.fn.expand("~/wiki")
  end,
  offset_encoding = "utf-16",
}
```

```lua
-- init.lua
vim.lsp.enable("zk-lsp")

vim.api.nvim_create_autocmd("LspAttach", {
  pattern = "*.typ",
  callback = function(ev)
    local client = vim.lsp.get_client_by_id(ev.data.client_id)
    if client and client.name == "zk-lsp" then
      vim.lsp.inlay_hint.enable(true, { bufnr = ev.buf })
    end
  end,
})
```

## Wiki Root Resolution

The wiki root is resolved in this order:

1. `--wiki-root` CLI flag
2. `WIKI_ROOT` environment variable
3. LSP `initializationOptions.root_dir` / `rootUri` / `workspaceFolders[0]`
4. `~/wiki` fallback

## CLI Reference

```
zk-lsp [OPTIONS] [COMMAND]

Commands:
  lsp        Start the LSP server on stdin/stdout [default]
  init       Initialise a new wiki in the current directory
  generate   Regenerate link.typ
  new        Create a new note and print its path to stdout
  remove     Delete a note and remove it from link.typ
  format     Read a note from stdin, write formatted content to stdout
  reconcile  Reconcile cross-file checkbox states [--dry-run]
  export     BFS context export for AI consumption [--depth N] [--inverse] [--simple]
  check      Graph integrity check: dead links + orphan notes
  notes      Output all notes' metadata/search records as JSON
  note-info  Output a single note's metadata as JSON
  metadata   Read or update one note's metadata record
  config     Inspect loaded zk-lsp configuration

Options:
  --wiki-root <PATH>   Override the wiki root directory
```

### Common examples

```bash
# Open a new note in Neovim
nvim $(zk-lsp new)

# Delete a note (removes file + link.typ entry)
zk-lsp remove 2602082037

# Format a note in-place (runs the hook pipeline)
zk-lsp format < note/2602082037.typ > /tmp/out.typ

# Propagate done-states across the wiki
zk-lsp reconcile --dry-run
zk-lsp reconcile

# Export context for an LLM (2 hops forward)
zk-lsp export 2602082037 --depth 2

# Export context (ancestors first — "what led to this idea?")
zk-lsp export 2602082037 --depth 2 --inverse

# Export only related note-info JSON objects
zk-lsp export 2602082037 --depth 2 --simple

# Check graph integrity (exits 1 on dead links)
zk-lsp check

# Read a combined note-info view as JSON (for scripts / AI agents)
zk-lsp note-info 2602082037

# Create or complete an existing note's metadata record without overwriting
# existing valid values
zk-lsp metadata create 2602082037

# Read one metadata record as JSON or TOML
zk-lsp metadata get 2602082037
zk-lsp metadata get 2602082037 --toml

# Update fields in a complete valid metadata record
zk-lsp metadata set 2602082037 checklist-status=todo
zk-lsp metadata set 2602082037 relation=archived relation-target='["2602082038"]'
zk-lsp metadata set 2602082037 user.priority=high keywords='["rust", "lsp"]'

# Reset named fields to their defaults; fields are not deleted
zk-lsp metadata reset 2602082037 checklist-status user.priority

# Read all notes' canonical metadata/search records in one process
zk-lsp notes --json

# Inspect the metadata schema implied by defaults + loaded config
zk-lsp config metadata fields --json --sources
zk-lsp config metadata defaults --toml
zk-lsp config metadata json-schema --json
```

### Metadata record commands

`zk-lsp metadata` operates on one note's metadata record in `metadata.toml`.
`create` is idempotent: it creates a missing record for an existing note or
fills missing fields from defaults while preserving existing valid values. It
fails instead of repairing invalid values, and it never creates metadata for a
missing `note/ID.typ`.

`get`, `set`, and `reset` require the current record to be complete and valid.
`set` applies all requested `KEY=VALUE` assignments together or rejects the
whole update. `reset` requires explicit field names and writes default values;
it does not delete fields. Format fields such as `schema-version` are managed by
`zk-lsp` and are not set or reset through this command family.

### Note template

```toml
[new_note]
template = """
#import "../include.typ": *
{{metadata}}
#show: zettel.with(metadata: zk-metadata)

= <{{id}}>

== References
"""
```

### Custom metadata fields

```toml
[[metadata.field]]
path    = "user.priority"   # must be user.*
kind    = "string"          # string | boolean | array-string
default = "normal"
```

### Metadata schema introspection

The `config metadata` command family prints the built-in metadata fields plus
custom fields from the merged user/project configuration:

```bash
zk-lsp config metadata fields       # JSON field specs (default output)
zk-lsp config metadata defaults     # JSON default metadata tree
zk-lsp config metadata json-schema  # JSON Schema draft 2020-12

zk-lsp config metadata fields --toml
zk-lsp config metadata defaults --toml
zk-lsp config metadata fields --sources
```

`--sources` adds a stable source list showing the user/project config paths and
whether each file was readable and parsed as TOML. For `json-schema`, this lives
under the `x-zk-lsp` extension key so the output remains a JSON Schema.

### Lua hooks

Running on `zk-lsp format` (e.g. on save).
The default pipeline includes built-in hooks for checklist normalisation, but you can add your own custom hooks or disable the defaults entirely.

```toml
disable_default_hooks = false   # set true to skip built-in checklist + relation hooks
[[hook]]
path = "~/.config/zk-lsp/hooks/my_hook.lua"
```

Example: [`examples/hooks/ensure_metadata_defaults.lua`](examples/hooks/ensure_metadata_defaults.lua)
backfills newly added `[[metadata.field]]` keys on save using their configured
defaults.

See [docs/lua-hooks.md](docs/lua-hooks.md) for the full hook API and examples.

### Reconcile rules

Running on `zk-lsp reconcile`, these rules propagate states across notes without manual Rust coding.
The default rules implement the task management workflow described below, but you can add your own rules or disable the defaults.

```toml
disable_default_reconcile_rules = false
[[reconcile.rule]]
path = "./reconcile/project_rules.lisp"
```

See [docs/reconcile-dsl.md](docs/reconcile-dsl.md) for the DSL reference and examples.

| Capability | Trigger |
|---|---|

| Diagnostics | `didOpen`, `didSave`, `didChangeWatchedFiles` |
| Code actions | On diagnostic ranges |
| References | `textDocument/references` |
| Workspace symbols | Symbol search by query string |
| Hover | `relation-target` IDs |
| Go-to-definition | `relation-target` IDs |

### LSP commands (`executeCommand`)

| Command | Arguments | Effect |
|---|---|---|
| `zk.newNote` | — | Create a note and notify with its URI |
| `zk.removeNote` | `id: string` | Delete a note |
| `zk.generateLinkTyp` | — | Regenerate `link.typ` |
| `zk.exportContext` | `id: string, depth?: number, inverse?: bool, simple?: bool` | Return Markdown context or simple note-info JSON |

## Diagnostics

| Condition | Severity |
|---|---|
| Dead `@ID` reference | Error |
| Dependency cycle | Error |
| Archived `@ID` reference | Warning |
| Legacy `@ID` reference | Info |
| Orphan note (no backlinks) | Hint |
| Schema error | Error / Warning |
| Non-leaf `@ID` item | Warning |

**Legacy suppression:** if a legacy reference is immediately followed by its
evolution ID on the same line (`@old @new`), the diagnostic is suppressed.

## A Quick Example: Life Cycle of a Task Note

An example which is also builtin in the default reconcile rules: using `@ID` references to manage task dependencies across notes.

Notes double as task management units via `@ID` checklist items:

```typst
- [ ] Task 1
  - [ ] Subtask 1.1 @2602082037   # done when referenced note is done
  - [ ] Subtask 1.2
```

Typical workflow:

1. `zk-lsp new` — note starts with `checklist-status = "none"`
2. Add `- [ ]` / `- [ ] @ID` entries
3. `zk-lsp format` — normalises checkboxes and updates `checklist-status`
4. `zk-lsp reconcile` — propagates states cross-file (topological order)
5. When superseded: set `relation = "legacy"` / `"archived"` and optionally
   add `relation-target = ["<successor-id>"]`

Cycles in task dependencies are a hard error — reported with Typst-style
source locations from both LSP diagnostics and `zk-lsp reconcile`.

## Inlay Hints

`@ID` references are rendered in two layers:

1. **Extmark conceal** — `@2602082037` → `@` (requires `conceallevel ≥ 2`)
2. **LSP inlay hint** — `@ Note Title`

The server sends `workspace/inlayHint/refresh` once the in-memory index is
built, so hints appear automatically without a manual reload.
